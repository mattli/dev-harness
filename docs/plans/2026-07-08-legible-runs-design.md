# Legible Runs — Design

**Date:** 2026-07-08
**Status:** Approved design, pending implementation plan
**Author:** Matt Li (product) + Claude (design)

## Problem

DevHarness produces run artifacts that a non-engineer product owner cannot read.
Two concrete failures, both hit in practice while reviewing the `mrbb5zo5`
CSV↔JSON run:

1. **Runs are unidentifiable.** Every run from every project lands in one flat
   `runs/<runId>/` folder, and `runId` is `Date.now().toString(36)` — an opaque
   timestamp like `mrbb5zo5`. There is no way to tell what a run was for, or
   which project it belonged to, without opening its files.

2. **The only human-facing artifact (`transcript.md`) is a raw engineering
   log.** It leads with plumbing (contract version counters, blank input/output
   digests, per-step token metering) instead of the story (what got built, how
   good it was, why it stopped). It is also actively misleading in places —
   e.g. `cost: $0.0000` on steps that did cost money but aren't metered.

The audience is the product owner, who directs the project but does not read the
implementation. The artifacts must answer "what happened?" in plain language.

## Goals

- A run's folder path alone tells you the project, the date, and what it was for.
- A new **summary** — plain English, ~6 lines — is the default thing you read.
- The existing **transcript** stops misleading and gains a light legend, but
  stays the detailed backup log.
- Works over SSH in a terminal (no GUI dependency).

## Non-Goals

- Capturing the AI's reasoning, inputs, or the generated code into the
  transcript (filling the blank `in:`/`out:` digests). That is a larger change
  to the trace-capture plumbing and is its own future project.
- Migrating or deleting existing runs. Old flat-layout runs (`e2e`, `mrbb5zo5`)
  stay exactly where they are; the new structure applies going forward only.
- Suggesting next actions in the summary. The summary states what happened;
  it does not prescribe what to do about it.

## Vocabulary

- **summary** — the plain-English recap (outcome, progress, quality, cost).
  "Recap" and "summary" are the same thing; the spec uses **summary**
  exclusively. It is not a standalone file — it is generated by the reporter and
  shown at the top of the transcript, in the terminal, and via `show`.
- **transcript** — the existing detailed step-by-step log (`transcript.md`).
- **the branch** — the actual generated code, which lives on a `run/...` git
  branch in the target project, not in the run folder.

## Design

### 1. Run folder structure and naming

Replace the flat `runs/<runId>/` with a two-level, human-readable layout:

```
runs/
  csv-json-converter/                    <- project slug (from --project path)
    2026-07-08-csv-json-converter/       <- <date>-<task-slug>
      transcript.md   (now opens with the summary; readable narrative below)
      state.json
      trace.jsonl
```

- **Project slug** — the basename of the `--project` path, slugified
  (lowercase, kebab-case). Zero user effort.
- **Date** — the run's start date, `YYYY-MM-DD`.
- **Task slug** — a short 3–4 word kebab-case title written by the AI planner
  (see §2).
- **Collision handling** — if `<project>/<date>-<task-slug>` already exists,
  append `-2`, `-3`, … so a run never overwrites a prior one.
- **Internal runId** — retained for uniqueness and as the identifier inside
  `state.json` / trace events / transcript header. It is no longer the folder
  name. (Kept so nothing downstream that keys on `runId` breaks.)

### 2. AI-written task label

The planner already reads the goal to produce the sprint plan. Add one field to
its structured output: `title` — a short kebab-case label (3–4 words).

Per the project convention "Don't parse model output by position — emit a marker
and key on it," the title is an explicit labeled field in the planner's
structured output, never scraped from prose. It is used for the task slug and as
the headline of the summary.

### 3. The reporter (summary engine)

A new module — the **reporter** — is the single source of the summary. It takes a
run's `state.json` and produces the plain-English text. It owns the translation
from internal codes to human language, most importantly the halt-reason lookup:

| Internal code           | Summary text                                      |
|-------------------------|---------------------------------------------------|
| `dollar-ceiling`        | Stopped early — hit the $X.XX spending limit.     |
| `wall-clock`            | Stopped early — hit the time limit.               |
| `max-iteration`         | Stopped — no improvement after the retry limit.   |
| `no-progress`           | Stopped — the score stopped improving.            |
| `evaluator-parse-error` | Stopped — an internal grading error.              |
| (status `passed`)       | Finished successfully — all stages passed.        |

Unknown/future codes degrade to a generic "Stopped — <code>." rather than
throwing, so a new halt reason never crashes the reporter.

Summary format (descriptive only, no recommendations):

```
csv-json-converter — 2026-07-08
Outcome:  Stopped early — hit the $6.80 spending limit
Progress: 3 of 4 stages finished
Quality:  scored 100, 98, 96 out of 100
Spent:    $6.80
Code:     saved on branch run/<slug>-<runId> in the target project
```

The summary is **not written as its own file.** It is generated by the reporter
and surfaced in three places, all from one source (no drift between them):

- **Top of `transcript.md`** — the summary block is embedded as the
  transcript's header, with the per-stage narrative (§4) flowing below it. So
  reading the transcript top-to-bottom gives the summary first, then detail.
- **Terminal** — the same text printed when a run finishes.
- **`dev-harness show <project> [run]`** — a new CLI command that regenerates
  the summary from a past run's saved `state.json` (so it works on any run
  without a standalone file). With only a project given, it shows the most
  recent run (or lists that project's runs if ambiguous — see Open Questions).

### 4. Transcript rewrite (readable narrative from existing data)

The transcript is rewritten from a raw phase-by-phase dump into a readable
per-stage narrative. It uses **only data the tool already records** — sprint
titles (from the plan), scores, costs, and tool-call counts. No new data is
captured (that is the "go deep" non-goal below).

The new transcript, per run:

- A header that **is the summary block** (§3), produced by the reporter — goal,
  outcome, progress, quality, cost. Same text the terminal and `show` print;
  one source.
- Below the header, one block **per stage (sprint)**, each showing:
  - The stage's plain-English title (from the plan's sprint title).
  - A status marker and the score (`✓ 98/100`) or `✗ stopped`.
  - The stage cost.
  - A one–two sentence narrative of what the stage did, derived from the
    tool-call counts — e.g. "created 8 files, ran 2 checks," "revised it 4
    times while running tests." Raw tool lists (`Write, Write, Bash, …`) are
    summarized into counts, not printed verbatim.
  - For a not-reached stage, why it never started (the halt reason).

Honesty fixes are folded in: no `cost: $0.0000` on unmetered steps, no blank
`in:`/`out:` lines. Phase jargon (NEGOTIATE, contract versions) is either
translated into the narrative ("agreed on the requirements") or dropped.

**Ceiling (explicitly out of scope here):** the transcript still cannot show
*why* a stage scored what it did, the actual generated code, or the AI's
reasoning — none of that is recorded today. Surfacing it is the "go deep"
project (see Non-Goals).

## Components and boundaries

- **reporter** (new): `state.json` → summary text. Pure function of run state;
  no I/O of its own beyond being handed the data. Independently testable with
  fixture states (including each halt code and the `passed` case).
- **run-path builder** (new/extracted): `(projectPath, date, title, existing
  dirs)` → collision-free run folder path. Pure, testable.
- **planner** (modified): emits `title` in structured output.
- **transcript renderer** (modified): honesty fixes + legend.
- **CLI** (modified): new `show` command; run-end prints the summary; `run`
  writes `transcript.md` (summary header + narrative). No standalone summary
  file.

## Testing

- **reporter**: unit tests per halt code + `passed` + unknown-code fallback;
  assert plain-English output and no "Next:" text.
- **run-path builder**: unit tests for slugification, date formatting, and
  collision suffixing.
- **planner**: test that `title` is present in structured output and used in the
  path.
- **transcript renderer**: test that `$0.0000`/blank-digest lines are gone, the
  per-stage narrative is present, and the header equals the reporter's summary
  output (one source, no drift).
- **boundary test**: per the project lesson "test guarantees at their real-I/O
  boundary," at least one test must run against the real filesystem to confirm
  `transcript.md` actually lands in the correct
  `runs/<project>/<date>-<task>/` path — not a mocked writer.

## Open questions

- `dev-harness show <project>` with multiple runs: default to most-recent, or
  print a list to pick from? (Lean: most-recent, with a `--list` to see all.)
- Slug source when `--project` is `.` or a path whose basename is generic
  (e.g. `src`): fall back to what? (Lean: parent dir name, else `project`.)

## Decisions log

- No standalone summary file: once the transcript opens with the summary and
  reads as a narrative, a separate `summary.md` is a duplicate to keep in sync.
  The summary lives at the top of the transcript, in the terminal, and via
  `show` (regenerated from `state.json`). The reporter remains the single
  engine that produces the summary text wherever it appears.
- No "Next:" recommendations — a wrong suggestion is worse than none; the owner
  decides next steps.
- Old runs left untouched (per "never delete" and to avoid a risky migration).
- Design doc kept in-repo (`docs/plans/`) not the vault, since it is tightly
  coupled to the code and versions alongside it.
