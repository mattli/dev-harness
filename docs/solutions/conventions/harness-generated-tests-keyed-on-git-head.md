# Harness-Generated Tests That Key on git HEAD Break Once the Sprint Is Committed

## Context
A dev-harness relocation sprint (`mrn0iav3`: move pure helpers from `bot.py` into
`session_state.py`) passed at score 96 with a green verifier — `124 passed, 1 skipped`.
But checking out the **committed** run branch and re-running the same suite gave
`19 failed, 105 passed`. The moved code was fine (an independent AST comparison showed
all 11 moved helpers/constants byte-for-byte identical); the *tests* broke.

The failing tests re-derive "the original" by shelling out to `git show HEAD:bot.py`
(verbatim-span checks, the manifest re-derivation, the diff-scope check, the
golden-from-source baseline). That is correct **only while the move is uncommitted** —
which is precisely how the harness verifier runs: it grades an uncommitted working-tree
diff against a `HEAD` still pointing at the pre-move commit, so `git show HEAD:bot.py`
returns the pre-move source that still contains the helpers. Once the sprint is
**committed**, `HEAD` advances to the post-move `bot.py` (helpers gone), and every
`HEAD`-keyed test fails on its own ground-truth lookup — e.g.
`AssertionError: load_profile not found in HEAD bot.py`.

## Guidance
**A harness-generated test that treats `git HEAD` as "the baseline" is coupled to the
verifier's uncommitted-diff execution model and is not portable as-is.** When harvesting
a run's tests into a real repo, split them by what they key on:

- **Behavior-pinning tests** (assert a helper's exact output on fixed inputs, check the
  module imports cleanly, measure branch coverage over the moved code) reference no git
  state. **Keep them as-is** — they stay green after commit and in CI.
- **HEAD-comparing tests** (verbatim-span equality vs `HEAD:bot.py`, manifest
  re-derivation from `HEAD`, `git diff HEAD` scope, golden regenerated by exec'ing
  `HEAD` spans) assume `HEAD` is the *pre-move* tree. Before harvesting, either **pin the
  baseline to a fixed pre-move commit** (or a committed fixture snapshot of the original
  source) instead of the moving `HEAD`, or **drop them** — their job (proving the move
  was verbatim) is a one-time migration check, not an ongoing invariant, so a
  behavior-pinning suite plus a reviewer's diff often covers the durable need.

The relocation's *ongoing* guarantee is "these helpers still behave as they did," which
the behavior-pinning tests express against a stable baseline. The verbatim-vs-`HEAD`
proof answers "was *this specific move* faithful," which stops being meaningful the
moment the move is history.

## Why This Matters
The failure looks alarming ("19 failed on the run branch") but is a false alarm about
the code and a true signal about the tests: a suite that is green inside the harness can
turn red in the repo it is harvested into, for a reason unrelated to the code under
test. Left unpinned, these tests fail in CI on the first commit after merge and get
written off as flaky or deleted wholesale — taking the portable behavior tests with
them. This is a sibling of [[test-guarantees-at-their-boundary]]: there, a guarantee
silently depended on a real-I/O boundary the tests stubbed; here, a test's own validity
silently depends on an implicit boundary condition — that `HEAD` is the pre-move tree —
which holds in the harness and not in the repo.

## When to Apply
- Deciding what to keep when porting a dev-harness run's output into a real codebase.
- Reviewing any generated verbatim-move / diff-scope / golden-from-source test: ask
  "what does it treat as the baseline, and is that baseline stable after commit?"
- Diagnosing a run branch whose suite is green in-run but red once checked out and
  committed — suspect `HEAD`-keyed tests before suspecting the code.

## Examples

**`mrn0iav3` file-by-file (the concrete harvest split):**

| Test file | Keys on `HEAD`? | Harvest action |
|-----------|:---------------:|----------------|
| `test_session_state_characterization.py` | no (43 output assertions) | **keep as-is** |
| `test_session_state_coverage.py` | no (branch coverage over moved spans) | **keep as-is** |
| `test_session_state_module.py` | no (import/structure checks) | **keep as-is** |
| `conftest.py`, `session_state_cases.py`, `_ss_coverage_driver.py` | no (fixtures / input corpus / driver) | **keep as-is** |
| `test_session_state_manifest.py` | **yes** (verbatim spans + manifest from `HEAD`) | **pin baseline or drop** |
| `test_session_state_golden.py` | **yes** (golden exec'd from `HEAD` spans) | **pin baseline or drop** |
| `test_session_state_botmove.py` | **yes** (`git diff HEAD` scope + reconstruction) | **pin baseline or drop** |

The three `HEAD`-keyed files are exactly the ones that fail on the committed branch
(13 + 5 + 1 = 19 failures); the rest stay green.

**Green in the harness's condition vs. red after commit:**
```
# pre-move HEAD (9ee4200) + the move as UNCOMMITTED changes  (how the verifier runs)
$ pytest -q  ->  124 passed, 1 skipped

# committed run branch (96e970a): HEAD is now the POST-move bot.py
$ pytest -q  ->  19 failed, 105 passed, 1 skipped
#   all 19 failures: "<helper> not found in HEAD bot.py"
```
(`9ee4200`/`96e970a` are commits in the run's target clone `voice-tutor-harness-run`,
not in this repo — they identify the pre-move base and the run tip of that run only.)
